0016: Analysis methodology: Every subsystem examined through the lens of a national security red team. 0017: 0.1 — EXISTENTIAL GAPS (Platform Will Not Launch)
0018: [GAP-01] asyncio Event Loop Architecture — Fatal Race Condition
0019: PROBLEM:
0020: The V2 AgentManager runs 250+ coroutines all sharing a single asyncio event loop. When the OllamaClient blocks on a 120s LLM timeout (circuit breaker), the ENTIRE event loop freezes — all 250 agents pause, WebSocket stops broadcasting, Redis subscriber dies. The platform appears operational but is completely unresponsive. This is an existential failure mode that will be hit within the first 30 minutes of operation.
0021: FIX:
0022: Use asyncio.wait_for() on ALL LLM calls with per-call timeouts (30s). Run a dedicated ThreadPoolExecutor for blocking I/O. Separate the Redis subscriber onto its own asyncio task with its own exception boundary. Use asyncio.shield() for critical broadcasts. Implement a watchdog coroutine that pings the event loop every 5 seconds and alerts if response > 2s.
0023: [GAP-02] No Process Isolation — Single Point of Catastrophic Failure
0024: PROBLEM:
0025: All 250+ agents, the API server, the Redis subscriber, and the WebSocket manager run in ONE Python process. If any agent hits an unhandled exception (e.g., rasterio segfault on malformed STAC data, or NumPy memory allocation error), the ENTIRE platform dies. On a national security platform, this is equivalent to a single point of failure in a nuclear command system.
0026: FIX:
0027: Architect as three separate processes: (A) backend/api_server.py — FastAPI + WebSocket only. (B) backend/agent_runner.py — All agent clusters, publishes to Redis only. (C) backend/fusion_runner.py — CrewAI fusion crew, triggered via Redis message. They communicate exclusively through Redis. Use supervisord or a Makefile with separate PIDs to monitor and restart each process independently.
0028: [GAP-03] Ollama Model Loading Has No Mutex — Guaranteed OOM on Startup
0029: PROBLEM:
0030: During startup, all 250 agents initialize simultaneously and the first N agents all attempt to call smart_generate() before any model is loaded. Ollama receives N simultaneous load requests for the same model. On an 8GB system this causes Ollama to attempt loading the model N times in parallel, triggering the OOM killer within seconds of startup.
0031: FIX:
0032: Implement a startup sequence gate: Agent clusters must not activate until OllamaModelManager.warmup() completes. warmup() loads the primary model ONCE, verifies it with a test prompt "reply: OK", then sets a global asyncio.Event called model_ready. All agents await model_ready before their first inference call. This single change prevents the most common crash on first launch.
0033: [GAP-04] WebSocket Connection Manager Has No Heartbeat — Silent Death
0034: PROBLEM:
0035: When a network interface flaps (Wi-Fi reconnect, VPN handshake), WebSocket connections enter a half-open state: the server believes the client is connected, the client receives no data. The broadcast() method iterates these zombie connections, send_json() hangs waiting for TCP ACK that never comes, the entire broadcast loop blocks for 60+ seconds. The HUD shows "CONNECTED" but receives zero events.
0036: FIX:
0037: Implement a ping/pong heartbeat: every 15 seconds, broadcast {"type":"ping","ts":timestamp}. Client must respond with {"type":"pong"} within 5 seconds or be forcibly disconnected. On the server, track last_pong per connection. If last_pong > 20s ago, remove from active_connections and close the socket. This is standard WebSocket production hardening.
0038: [GAP-05] No Graceful Shutdown — Data Corruption on Ctrl+C
0039: PROBLEM:
0040: When the operator presses Ctrl+C or runs "make stop", all processes receive SIGTERM simultaneously. SQLite WAL checkpoint is never completed, ChromaDB write buffers are not flushed, in-flight agent results are lost, and Redis pub/sub messages are dropped. On subsequent restart, SQLite may enter recovery mode (slow) or ChromaDB index may be corrupted (catastrophic).
0041: FIX:
0042: Register signal handlers for SIGTERM and SIGINT. On shutdown: (1) stop accepting new agent tasks, (2) wait for in-flight tasks with timeout=10s, (3) flush ChromaDB client, (4) run sqlite3 PRAGMA wal_checkpoint(TRUNCATE), (5) publish {"type":"shutdown"} to Redis so clients show "PLATFORM OFFLINE", (6) then exit. Use a shutdown_event asyncio.Event that all loops check.

0081: PROBLEM:

0085: [GAP-14] CrewAI Agents Have No Token Budget — Context Window Overflow
0086: PROBLEM:
0087: CrewAI agents chain multiple LLM calls and accumulate conversation history. After 3-4 tool calls, the context window for llama3.2:1b (128K tokens but only ~2K practical for 8GB inference) overflows. The model begins hallucinating, confabulating threat intelligence, and repeating itself in infinite loops. This has been observed in practice and is guaranteed to occur within 2 hours of operation.
0088: FIX:
0089: Set explicit max_tokens=512 for ALL individual agent LLM calls in CrewAI. Set max_iter=3 for all CrewAI agents (prevents infinite tool-use loops). Log truncation events as WARN so the operator knows the model is hitting its limits.
0090: [GAP-15] SQLite Concurrent Write Contention Despite WAL Mode
0091: PROBLEM:
0092: WAL mode helps readers but does not solve writer contention. With 250 agents all completing their 30-second cycles within the same second (due to the synchronous gather() pattern), you get a thundering herd of 250 simultaneous SQLite writes. WAL handles this but causes checkpoint storms every ~1000 pages, spiking CPU to 100% and triggering macOS thermal throttling — the "jet engine" the spec warns about.
0093: FIX:
0094: Implement a write buffer: agents do NOT write to SQLite directly. They push AgentResult objects to a dedicated write_queue (asyncio.Queue). A single DatabaseWriter coroutine drains this queue and performs bulk INSERTs using executemany() in batches of 25, every 5 seconds. This converts 250 individual writes into 10 batch operations per cycle, reducing SQLite I/O by 96%.
0095: [GAP-16] No Input Validation on API Endpoints — SQL Injection and Memory Bombs
0096: PROBLEM:
0097: The /events endpoint accepts query parameters (limit, offset, cluster, threat_level) that are concatenated directly into SQLite queries without parameterization. Additionally, no maximum limit is enforced — a client can request limit=1000000, causing SQLite to attempt to load 1M rows into memory, OOM-killing the API process.
0098: FIX:
0099: Use Pydantic query parameter models with strict validation: limit: int = Field(default=50, ge=1, le=200). ALWAYS use parameterized queries: cursor.execute("SELECT * FROM threat_events WHERE cluster=?", (cluster,)) — NEVER f-string SQL. Add a response size guard: if len(results) == limit, set a response header X-Truncated: true so clients know more data exists.
0100: [GAP-17] Integrity Hash Is Trivially Forgeable
0101: PROBLEM:
0102: The V2 integrity_hash is SHA-256 of (event_id + title + timestamp). These are ALL values that the platform itself generates and controls. There is no secret key component. Anyone with read access to the SQLite database can generate a valid integrity_hash for any forged record. The hash provides zero security guarantee — it only detects accidental data corruption, not adversarial tampering.
0103: FIX:
0104: Use HMAC-SHA256 with a platform secret key: hmac.new(SECRET_KEY, (event_id+title+ts).encode(), sha256).hexdigest(). The SECRET_KEY is a 32-byte random value generated on first startup and stored in ~/shakti/.platform_secret (chmod 600, never in Git). Without the key, hashes cannot be forged. This elevates integrity verification from error-detection to tamper-evidence.
0105: [GAP-18] Darkweb Forum Simulator Uses Predictable Patterns — AI Overfit
0106: PROBLEM:
0107: The SIGINT simulator generates fake darkweb text using fixed templates ("phishing a healthcare intranet", "free SSL certs for malicious domains"). After 10 training cycles, the LLM learns to flag EXACTLY these phrases and nothing else. Novel threat language (new slang, coded references, encrypted steganographic messages) will score 0 threat probability because the model has overfit to the simulator vocabulary.
0108: FIX:
0109: Implement a paraphrase rotation layer: store 50 base threat scenarios, generate 10 paraphrases of each using the local LLM during startup (one-time cost). Rotate through 500 distinct phrasings. Add 5 "false positive" scenarios (legitimate security research discussions) to train the model to avoid over-flagging. This creates a more robust training distribution without any external data dependency.

0131: [GAP-23] No Data Retention and Purge Policy — Disk Exhaustion in 60 Days

0141: [GAP-25] No TLS on WebSocket — Cleartext Intelligence in Transit
0142: PROBLEM:

0146: [GAP-26] Event UUID Uses Python uuid.uuid4() — No Temporal Ordering
0147: PROBLEM:
0148: The event_id uses random UUID4, which has no temporal ordering. When reconstructing attack timelines from the database (critical for post-incident analysis), you must rely on the timestamp_iso string sort, which is fragile if timestamps have any clock skew or are generated slightly out of order between agents. UUID4-based IDs cannot be used for efficient range queries.
0149: FIX:
0150: Use UUIDv7 (time-ordered UUID): the first 48 bits are a Unix timestamp in milliseconds. This makes event_ids sortable, timeline-reconstructible from the ID alone, and enables efficient range queries (WHERE event_id BETWEEN uuid7(t1) AND uuid7(t2)). Install with pip install uuid7. All new events: from uuid7 import uuid7; event_id = str(uuid7()).

0171: [GAP-31] The 8GB Constraint Is Treated as a Hard Limit, Not a Dynamic System
0172: PROBLEM:
0173: The V2 architecture treats "8GB available" as fixed. In reality, macOS unified memory is dynamic: the GPU may need 1-2GB for a video call, another app may spike to 500MB, Time Machine may start a backup. The static thresholds (5.0GB throttle, 6.5GB critical) become dangerously wrong in these scenarios. The platform needs to respond to the ACTUAL available memory, not a theoretical available memory.
0174: FIX:

When CRITICAL: immediately send SIGSTOP to the Ollama process (pauses inference without killing the model) and SIGCONT when pressure resolves.
